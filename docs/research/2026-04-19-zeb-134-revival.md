# ZEB-134 revival — context for a future reader

**Date:** 2026-04-19
**Branch landing:** `zeblith/zeb-139-kl-retrofit-impl` (this PR)
**Original commit:** `3f7cae33` on `zeblith/zeb-134-skip-to-logit` (2026-04-17)

## Why the original commit message reads like a negative result

The cherry-picked ZEB-134 commit message documents an experiment that
*failed*: under CE-only training, the Skip-to-Logit Engram Router hit
the maximum-entropy attractor (`engram_logit_entropy = log(32000)`,
`α ≈ 0.17`) and produced `Δ-diff = +0.0002` against the ι₂ baseline's
`+0.0003`. Per the decision table at the time, the verdict was that
the "40M-capgap steelman is confirmed at this scale" and the
recommendation was to "move to either larger student or
teacher-architecture-match experiments."

That branch was authored on 2026-04-17 and was never opened as a PR.
It sat unmerged on origin while subsequent work (ZEB-130 vcontrast /
qdiv, ZEB-128 consolidation, ZEB-138 same-arch teacher, ZEB-119/254
oracle scaffolding) merged independently. ZEB-138 followed the
"teacher-architecture-match" recommendation directly.

## Why we're reviving it now

The 2026-04-18 Gemini Deep Research analysis (saved to
`docs/research/2026-04-18-post-zeb138-path-research-findings.md` on
the ZEB-138 branch) reframed the negative ZEB-134 result. The
attractor isn't evidence that the architecture is broken — it's
gradient descent **autonomously inventing label smoothing** when the
router signal is uninformative under cross-entropy. `P_router` is
driven to `1/|V|` because that minimizes the expected CE penalty of
mixing arbitrary noise into `P_LM`, and `α ≈ 0.17` happens to be the
empirically optimal label-smoothing factor for the FineWeb-Edu-POC
dataset given the base 40M LM's confidence calibration.

Memory Decoder (Cao et al., NeurIPS 2025, arXiv:2508.09874) escapes
this trap by adding a `KL(P_router || P_teacher)` term to the
training objective. The dense, content-rich KL gradient pre-aligns
`P_router` to the teacher's distribution **before** CE pressure kicks
in to flatten it.

[ZEB-139](https://linear.app/zeblith/issue/ZEB-139) tests whether
adding that KL term to the existing ZEB-134 architecture breaks the
attractor on the cross-arch TinyLlama setup that ZEB-136 first
observed it on. ZEB-139's experimental design — fingerprint metrics,
4-cell matrix, verdict table — is **predicated on ZEB-134's
architecture being present**. The spec at
`docs/superpowers/specs/2026-04-18-zeb-139-kl-retrofit-design.md`
references `engram_logits`, `W_align`, and `engram_logit_entropy` as
if they exist on main; without ZEB-134 they don't.

So the cherry-pick is "completing what should have shipped" —
unblocking a branch that was shelved for the right reason at the time
(no point landing failed code without a follow-up plan), but whose
follow-up plan now exists in the form of ZEB-139.

## What this means for the original commit message

The negative-result verdict in the cherry-picked commit message
**still stands** for the standalone ZEB-134 experiment. CE-only
training with the Skip-to-Logit Router does hit the attractor at
40M. That isn't being relitigated. The second commit on this branch
adds the KL+CE pathway that ZEB-139 hypothesizes will break the
attractor; the 4-cell matrix run that follows will produce the actual
ZEB-139 result.
